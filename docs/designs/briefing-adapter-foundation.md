## Vision

### 10x Check

当前版本用模拟数据生成 JSON 简报。10 倍版本是一个真正能用的信息处理管道：
- 接入 RSSHub 拿到真实 RSS 数据流
- 通过 Adapter Protocol 让社区能贡献新的数据源适配器
- JSON Schema 运行时校验保证数据质量
- 多种输出格式（JSON + Markdown）
- Source health check 让运维可观测
- 缓存层避免重复处理
- 15-20 个高质量信息源条目

### Platonic Ideal

用户的 AI 助手早上自动调用 News Report，传入偏好。15 秒后收到精炼简报：
5 条最相关信息，每条都有推荐理由。来自播客、RSS、阅读列表多个渠道，
已自动去重、评分、排序。Adapter 是插件化的，社区贡献新 adapter 就像
给 RSSHub 加路由一样简单。

## Implementation Sequence

实现顺序很重要。先做核心，再做外围：

1. **Adapter Protocol** — 先定义接口，后面的都依赖它
2. **RSSHub 真实适配器** — 第一个实现协议的真实 adapter
3. **JSON Schema 运行时校验** — 有真实数据后校验才有意义
4. **Briefing 缓存** — 有真实网络调用后缓存才有意义
5. **Source Health Check** — 有真实 adapter 后健康检查才有意义
6. **Markdown 输出** — 独立功能，可随时做
7. **sources.json 扩展** — 纯数据工作，可随时做

## Concrete Interface Design

### Adapter Protocol

```python
from typing import Protocol, runtime_checkable
from datetime import datetime

@runtime_checkable
class SourceAdapter(Protocol):
    source_id: str

    def fetch(self, topic: str, summary_style: str, now: datetime | None = None) -> list[dict]:
        """获取候选内容。同步调用。返回 candidate dict 列表。"""
        ...

    def ping(self) -> bool:
        """检查数据源是否可达。超时 5 秒返回 False。"""
        ...
```

返回的 candidate dict 格式：
```python
{
    "id": str,             # 唯一 ID
    "title": str,          # 标题
    "source": str,         # source_id
    "source_name": str,    # 显示名
    "content_type": str,   # article/podcast/transcript 等
    "url": str,            # 原始链接
    "published_at": datetime,  # 发布时间（UTC）
    "tags": list[str],     # 标签
    "angles": list[str],   # 角度/视角
    "summary": str,        # 摘要
    "source_meta": dict,   # sources.json 中的源信息
}
```

### Adapter 注册机制

`build_adapter_registry()` 改为按 source 的 `category` 和 `id` 自动选择 adapter：
- 有真实 adapter 的源 → 用真实 adapter
- 没有真实 adapter 的源 → fallback 到 StaticSourceAdapter（mock）
- 新增源只需要加 catalog 条目，不强制要求有 adapter

### RSSHub Adapter 实现要点

- 使用 `httpx`（async-capable HTTP 库）+ `feedparser`（RSS 解析）
- 默认实例：`https://rsshub.app`，可通过环境变量 `RSSHUB_BASE_URL` 覆盖
- 超时：5 秒连接超时，10 秒读取超时
- 失败处理：超时/网络错误 → 返回空列表 + 日志警告，不中断整个 briefing
- 不支持认证（RSSHub 公共实例无需认证）

### JSON Schema 校验策略

- `validate_request()` 替换为 `jsonschema.validate(data, schema)` 调用
- `validate_sources()` 替换为遍历 + schema 校验
- 校验发生在 **输入侧**（request + sources），输出侧暂不校验
- 依赖：`jsonschema` 库，加入 `pyproject.toml` dependencies

### Markdown 输出

新增函数 `format_briefing_markdown(briefing: dict) -> str`，独立于 `generate_briefing()`。
CLI 加 `--format {json,markdown}` 参数，默认 json。

输出格式示例：
```markdown
# News Briefing - 2026-03-31

## 1. [标题](url)
- 来源：Podwise CLI | 类型：podcast
- 评分：0.86
- **为什么推荐：** 强话题匹配 + 匹配显式偏好 agent tools
- 摘要内容...

## 2. ...
```

### 新增依赖

`pyproject.toml` 的 `dependencies` 从空列表变为：
- `httpx` — HTTP 客户端（RSSHub adapter）
- `feedparser` — RSS/Atom 解析
- `jsonschema` — JSON Schema 校验

### sources.json 扩展标准

新增条目必须满足：
- 所有 schema 必填字段完整
- `agent_friendly` 标注有依据
- `last_verified` 在 30 天内
- 新增条目不需要配套 adapter，但需标注 `has_adapter: false`

优先补充的源类型：Hacker News API、GitHub Trending、Arxiv RSS、
Newsletter 工具（Kill the Newsletter）、YouTube 转录工具等。

## Failure Modes

| 场景 | 处理方式 | 用户看到的 |
|------|----------|-----------|
| RSSHub 实例不可达 | 返回空列表 + 日志警告 | briefing 中不包含该源，coverage 显示 0 items |
| RSSHub 返回畸形 XML | feedparser 容错解析 + 跳过无法解析的条目 | 正常输出，条目数可能少于预期 |
| 缓存文件损坏 | 删除损坏文件，重新请求 | 正常输出，略慢 |
| JSON Schema 校验失败 | 抛出 ValueError + 具体字段信息 | CLI 输出错误信息 |
| ping() 超时 | 5 秒后返回 False | health check 显示该源不可用 |

## Accepted Scope

- RSSHub 真实适配器：接入公共实例，拿到真实 RSS 数据
- Adapter Protocol：定义正式的适配器接口协议（见上方具体设计）
- JSON Schema 运行时校验：用 jsonschema 库替换手写校验
- Markdown 输出：新增 format_briefing_markdown() 函数 + CLI --format 参数
- Source Health Check：adapter 加 ping() 方法，coverage 加可用状态
- Briefing 缓存：基于请求参数 hash 的文件级缓存，TTL 15 分钟
- sources.json 扩展：补充到 15-20 条高质量条目
